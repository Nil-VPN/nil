# nil-ios

The iOS engine for NIL VPN: a **C-ABI staticlib** linked into a `NEPacketTunnelProvider` Network
Extension. iOS has no TUN fd — packets flow over the extension's `NEPacketTunnelFlow`, so the pump
is callback-driven (`nil_ingest_packets` in, the write callback out). The MASQUE transport is
reused unchanged; this crate deliberately does **not** depend on `nil-datapath` (which needs
`tun-rs`). Routing/DNS/MTU are the extension's `NEPacketTunnelNetworkSettings`, and the extension's
own sockets bypass the tunnel by default — so there is no `protect()`/NetControl.

Identity never reaches the extension — only a node endpoint + optional pinned measurement; the
unlinkable token is redeemed in the container app.

## Status (verified)

- ✅ **BoringSSL / quiche cross-compile for `aarch64-apple-ios`** (Xcode 26.5 accepted it; no
  bitcode failure). The iOS analog of the Android gate.
- ✅ **`libnil_ios.a` cross-compiles** for iOS-arm64.
- ✅ C header generated by cbindgen (`include/nil_ios.h`) — the FFI Swift imports.
- ✅ Reference `NEPacketTunnelProvider` written (`apple/PacketTunnelProvider.swift`).

## Build

```sh
# host (validates the FFI + regenerates include/nil_ios.h)
cargo build -p nil-ios

# device slice
export IPHONEOS_DEPLOYMENT_TARGET=15.0 CMAKE_POLICY_VERSION_MINIMUM=3.5
cargo build -p nil-ios --release --target aarch64-apple-ios
cargo build -p nil-ios --release --target aarch64-apple-ios-sim   # simulator slice (rustup target add first)
# package both slices:
xcodebuild -create-xcframework \
  -library target/aarch64-apple-ios/release/libnil_ios.a -headers crates/nil-ios/include \
  -library target/aarch64-apple-ios-sim/release/libnil_ios.a -headers crates/nil-ios/include \
  -output NilIOS.xcframework
```

## Remaining (the documented next step)

1. `cd client && pnpm tauri ios init` — generates `gen/apple/` (Xcode project).
2. Add a **PacketTunnel** appex target; drop in `apple/PacketTunnelProvider.swift`, a bridging
   header that `#import "nil_ios.h"`, and link `NilIOS.xcframework`. The container app does NOT
   link the staticlib.
3. App Group (`group.com.nilvpn.client`) for app↔extension config; the app writes the node
   endpoint + measurement into `providerConfiguration`.
4. Container-app UI builds + runs in the **Simulator** (UI only).

## Hard blockers (cannot be done headlessly — founder/device-bound)

- **Apple Developer ORG enrollment** + the **`packet-tunnel-provider` entitlement** (Apple email
  review, weeks) — without it the appex cannot be signed/run.
- **A physical iOS device** — packet tunnels do **not** run in the Simulator, so the datapath
  cannot be validated here at all.
- Real code-signing / provisioning profiles.
- Harden `readPackets` pointer lifetimes in the Swift before shipping (the reference copies
  synchronously, which is correct because `nil_ingest_packets` copies, but review under ASan).
