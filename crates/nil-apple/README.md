# nil-apple

The Apple engine for NIL VPN: a **C-ABI staticlib** linked into a `NEPacketTunnelProvider` Network
Extension — **shared by the iOS app extension and the macOS system extension** (same provider, same
Rust engine). Neither has a TUN fd — packets flow over the extension's `NEPacketTunnelFlow`, so the
pump is callback-driven (`nil_ingest_packets` in, the write callback out). The MASQUE transport is
reused unchanged; this crate deliberately does **not** depend on `nil-datapath` (which needs
`tun-rs`). Routing/DNS/MTU are the extension's `NEPacketTunnelNetworkSettings`, and the extension's
own sockets bypass the tunnel by default — so there is no `protect()`/NetControl.

Identity never reaches the extension — only a node endpoint, optional pinned measurement, and the
per-connection Privacy Pass grant (token + freshness nonce); the unlinkable token is redeemed in the
container app.

## Status (verified)

- ✅ **BoringSSL / quiche cross-compile for `aarch64-apple-ios`** (Xcode 26.6 accepted it; no
  bitcode failure). The iOS analog of the Android gate. Re-verified on the current tree.
- ✅ **`libnil_apple.a` cross-compiles** for iOS-arm64.
- ✅ C header generated by cbindgen (`include/nil_apple.h`) — the FFI Swift imports.
- ✅ Reference `NEPacketTunnelProvider` written (`apple/PacketTunnelProvider.swift`).
- ✅ **iOS Tauri plugin written** (`apple/NilVpnPlugin.swift`) — the container-app counterpart of
  Android's `NilVpnPlugin.kt`. Implements the five `nil-vpn` commands the WebView calls
  (`prepareVpn`/`startVpn`/`statusVpn`/`stopVpn`/`openVpnSettings`) over `NETunnelProviderManager`,
  forwarding the attested args into the appex's `providerConfiguration`. **API-verified against the
  `tauri 2.11.3` Swift sources** (`Plugin`/`Invoke`/`parseArgs`/`resolve`/`reject`, `JsonObject =
  [String: Any?]`, and the `ios_plugin_binding!` → `@_cdecl("init_nil_vpn_plugin") -> Plugin` FFI
  contract); parses clean with `swiftc`.
- ✅ **Rust registration wired** — `client/src-tauri/src/lib.rs` uses
  `tauri::ios_plugin_binding!(init_nil_vpn_plugin)` + `register_ios_plugin` (replacing a
  wrong-signature hand-written `extern "C"` block that would not have linked).
- ✅ **iOS appex Info.plist + entitlements** added (`apple/PacketTunnel-iOS-Info.plist`,
  `apple/PacketTunnel-iOS.entitlements`, `apple/App-iOS.entitlements`) — the iOS app-extension flavor
  (`packet-tunnel-provider`, `XPC!`, principal class) distinct from the macOS system-extension ones.
- ✅ **`pnpm tauri ios init` generates the Xcode project** (`gen/apple`, via an XcodeGen
  `project.yml`) on Xcode 26.6 — so the appex target can be added declaratively.

The iOS client is now **code-complete**; what remains is device/Xcode-bound (signing + a physical
device), documented in `DEVICE_VERIFY.md`.

## Build

```sh
# host (validates the FFI + regenerates include/nil_apple.h)
cargo build -p nil-apple

# device slice
export IPHONEOS_DEPLOYMENT_TARGET=15.0 CMAKE_POLICY_VERSION_MINIMUM=3.5
cargo build -p nil-apple --release --target aarch64-apple-ios
cargo build -p nil-apple --release --target aarch64-apple-ios-sim   # simulator slice (rustup target add first)
# package both slices:
xcodebuild -create-xcframework \
  -library target/aarch64-apple-ios/release/libnil_apple.a -headers crates/nil-apple/include \
  -library target/aarch64-apple-ios-sim/release/libnil_apple.a -headers crates/nil-apple/include \
  -output NilApple.xcframework
```

## Remaining (device/Xcode-bound — needs a Mac with Xcode, an enrolled Apple org, and a device)

The Rust/Swift sources are done; these steps wire them into the generated Xcode project and sign for
a device. They can't be verified headlessly (no signing/device here), so they're the device handoff —
see `DEVICE_VERIFY.md` for the exact `project.yml` stanzas and commands.

1. `cd client && pnpm tauri ios init` — generates `gen/apple/` (XcodeGen `project.yml`). **Done once.**
2. **Compile the plugin into the app:** add `apple/NilVpnPlugin.swift` to the app target's sources and
   make the `Tauri` Swift package available to it (it `import Tauri`), so `init_nil_vpn_plugin` links
   into `libapp.a`'s `register_ios_plugin` call.
3. **Add the PacketTunnel appex target** to `project.yml`: principal class
   `apple/PacketTunnelProvider.swift` + a bridging header that `#import "nil_apple.h"`, link the
   `ios-arm64` slice of `NilApple.xcframework` (build via `NIL_APPLE_WITH_IOS=1 client/apple/build-engine.sh`),
   `info: PacketTunnel-iOS-Info.plist`, `entitlements: PacketTunnel-iOS.entitlements`. The container
   app embeds the appex and does NOT link the staticlib itself.
4. **Entitlements:** merge `App-iOS.entitlements` (the `packet-tunnel-provider` capability + the
   `group.com.nilvpn.client` App Group) into the generated app entitlements. App + appex share the group.
5. Set `DEVELOPMENT_TEAM` (untracked local xcconfig), sign, build to a device, run the
   `DEVICE_VERIFY.md` leak/kill-switch/memory checks.

## Hard blockers (cannot be done headlessly — founder/device-bound)

- **Apple Developer ORG enrollment** + the **`packet-tunnel-provider` entitlement** (Apple email
  review, weeks) — without it the appex cannot be signed/run.
- **A physical iOS device** — packet tunnels do **not** run in the Simulator, so the datapath
  cannot be validated here at all.
- Real code-signing / provisioning profiles.
- Harden `readPackets` pointer lifetimes in the Swift before shipping (the reference copies
  synchronously, which is correct because `nil_ingest_packets` copies, but review under ASan).
